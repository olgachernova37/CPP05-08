Вправа 02: Визначення реального типу (Identify real type)
Каталог для здачі: ex02/

Файли для здачі: Makefile, *.cpp, *.{h, hpp}

Заборонено: <typeinfo> (бібліотека std::typeinfo)

Реалізуй клас Base, який має лише публічний віртуальний деструктор.

Створи три порожні класи A, B та C, які публічно наслідуються від Base.

(Ці чотири класи не обов'язково реалізовувати в Православній Канонічній Формі (OCF)).

Реалізуй такі функції:

Base * generate(void);

Випадковим чином створює екземпляр класу A, B або C та повертає його як вказівник на Base. Для генерації випадкового вибору можеш використовувати будь-який зручний спосіб.

void identify(Base* p);

Виводить на екран фактичний тип об'єкта, на який вказує p: "A", "B" або "C".

void identify(Base& p);

Виводить на екран фактичний тип об'єкта, на який посилається p: "A", "B" або "C". Використовувати вказівники всередині цієї функції суворо заборонено.

Включення заголовного файлу <typeinfo> заборонено.

Напиши програму для тестування, щоб переконатися, що все працює належним чином.

📚 Теорія, яка тобі знадобиться
Це завдання повністю присвячене динамічному приведенню типів за допомогою dynamic_cast та концепції RTTI.

1. Що таке dynamic_cast?
У C++ існує чотири типи кастингу. У попередній вправі ми розбирали reinterpret_cast. Тут же панує dynamic_cast.

dynamic_cast використовується виключно для безпечного приведення вказівників або посилань по ієрархії наслідування (найчастіше — вниз, від базового класу до похідного, що називається downcasting).

⚠️ Важливо: dynamic_cast працює лише в тому випадку, якщо базовий клас є поліморфним (тобто містить хоча б одну віртуальну функцію). Саме тому в завданні вимагається зробити деструктор класу Base віртуальним (virtual ~Base()).

2. Як працює dynamic_cast з вказівниками (Base*)?
Коли ти намагаєшся привести вказівник базового класу до вказівника похідного класу (наприклад, Base* до A*):

Якщо об'єкт дійсно є типом A, кастинг проходить успішно і повертає правильну адресу.

Якщо об'єкт насправді іншого типу (наприклад, B), dynamic_cast повертає NULL (або nullptr у сучасних стандартах).

Це дає нам чудовий інструмент для перевірки:

C++
if (dynamic_cast<A*>(p) != NULL) {
    std::cout << "A" << std::endl;
}
3. Як працює dynamic_cast з посиланнями (Base&)?
Це найцікавіша частина завдання. Посилання в C++ не можуть бути "нульовими" (NULL). Тому, якщо спроба привести посилання за допомогою dynamic_cast зазнає невдачі:

Компілятор не може повернути NULL.

Замість цього він генерує виняток (exception) типу std::bad_cast.

Оскільки нам заборонено використовувати вказівники всередині identify(Base& p), ми повинні використовувати конструкцію try-catch для обробки цих винятків. Якщо кастинг ламається — ми йдемо в блок catch і пробуємо наступний тип.

C++
try {
    A& a = dynamic_cast<A&>(p);
    std::cout << "A" << std::endl;
    (void)a; // Уникаємо попередження про невикористану змінну
} catch (const std::exception& e) {
    // Якщо це не А, переходимо до перевірки B...
}
4. Генерація випадкових об'єктів
Для реалізації функції generate() тобі знадобиться генератор випадкових чисел. У C++98 це зазвичай робиться за допомогою функцій rand() та srand(), які знаходяться в <cstdlib>, а ініціалізувати генератор поточним часом можна через time(NULL) з <ctime>.

Наприклад:

C++
#include <cstdlib>
#include <ctime>

// У main() один раз викликаємо:
std::srand(std::time(NULL));

// У функції generate():
int random_choice = std::rand() % 3; // Поверне 0, 1 або 2
💡 Чому заборонено <typeinfo>?
Заголовок <typeinfo> дозволяє використовувати оператор typeid, який міг би повернути назву класу буквально в один рядок кода: typeid(*p).name().
Заборона цього заголовка змушує тебе вручну написати логіку перевірки типів через dynamic_cast, що і є головною метою цієї вправи — навчитися працювати з безпечним приведенням типів "руками".

🔍 Ключові моменти для захисту, які ти маєш розуміти:
Чому в identify(Base& p) ми ловимо std::exception? У C++98 dynamic_cast для посилань викидає виняток типу std::bad_cast. Оскільки він наслідується від базового класу std::exception, ловити const std::exception& є абсолютно безпечним і правильним за стандартом.

Навіщо (void)a; у блоках try?
Компілятори з прапорцями -Wall -Wextra -Werror (які обов'язкові в 42) видадуть помилку про те, що створена змінна a ніде не використовується. Цей трюк просто "заспокоює" компілятор.

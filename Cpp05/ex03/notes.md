Тепер, коли у вас є базові форми, настав час створити ще кілька таких, які дійсно щось роблять.

У всіх випадках базовий клас AForm має бути абстрактним класом, і тому його слід перейменувати на AAForm. Майте на увазі, що атрибути форми повинні залишатися приватними (private) і належати базовому класу.

Додайте наступні конкретні (похідні) класи:

ShrubberyCreationAForm (Форма створення чагарників): Необхідні грейди: підписання — 145, виконання — 137.
Створює файл <target>_shrubbery у робочому каталозі та записує туди ASCII-дерева.

RobotomyRequestAForm (Форма запиту на роботомію): Необхідні грейди: підписання — 72, виконання — 45.
Видає звуки свердління (дрилі), а потім повідомляє, що <target> успішно роботомізовано у 50% випадків. В іншому випадку повідомляє, що роботомія провалилася.

PresidentialPardonAForm (Форма президентського помилування): Необхідні грейди: підписання — 25, виконання — 5.
Повідомляє, що Зафод Біблброкс (Zaphod Beeblebrox) помилував ціль (<target>).

Усі вони приймають лише один параметр у своєму конструкторі: ціль (target) форми. Наприклад, "home", якщо ви хочете "посадити чагарник" вдома.

Тепер додайте функцію-член execute(Bureaucrat const & executor) const до базової форми (AAForm) та реалізуйте функцію для виконання дії форми у конкретних класах. Ви повинні перевірити, що форма підписана, а грейд бюрократа, який намагається виконати форму, є достатньо високим. Інакше викидайте відповідний виняток.

Перевіряти ці вимоги у кожному конкретному класі чи у базовому класі (а потім викликати іншу функцію для виконання форми) — вирішувати вам. Проте один із цих способів є значно елегантнішим за інший.

Зрештою, додайте функцію-член executeAForm(AAForm const & AForm) const до класу Bureaucrat. Вона повинна спробувати виконати форму. У разі успіху виведіть щось на кшталт:
<bureaucrat> executed <AForm>
Якщо ні — виведіть чітке повідомлення про помилку.

Теоретичний розбір завдання
Ця вправа переводить роботу з ООП на рівень архітектурного проєктування, фокусуючись на абстракції, поліморфізмі та правильному розподілі обов'язків між класами.

1. Абстрактні класи та чисто віртуальні функції (Pure Virtual Functions)
Базовий клас AForm перейменовується на AAForm (префікс 'A' традиційно означає "Abstract").

Що таке абстрактний клас? Це клас, об'єкти якого не можна створити безпосередньо (код AAForm myAForm; викличе помилку компіляції). Він служить виключно шаблоном (інтерфейсом) для нащадків.

Як зробити клас абстрактним у C++? Для цього потрібно оголосити хоча б одну чисто віртуальну функцію (pure virtual function) за допомогою синтаксису = 0:

C++
virtual void executeAction() const = 0; // Чисто віртуальна функція
Кожен конкретний клас-нащадок (Shrubbery..., Robotomy...) зобов'язаний перевизначити (override) цю функцію і реалізувати власну логіку (малювання дерев, роботумію тощо).

2. Інкапсуляція базового класу та робота з target
У темі зазначено: "атрибути форми повинні залишатися приватними і належати базовому класу".

Важливе правило: Нащадки не мають прямого доступу до приватних (private) полів AAForm. Тому, щоб дізнатися ім'я, статус підпису або грейди всередині похідних класів, ви повинні використовувати публічні або захищені геттери (getName(), getIsSigned() тощо), які ви написали в AAForm.

Поле target (ціль) є унікальним для кожного типу форми, але оскільки воно фігурує скрізь, його можна зберігати як у базовому класі AAForm, так і в самих нащадках (зазвичай зручніше передавати його в конструктор базового класу або зробити захищений геттер для нього).

3. Патерн проектування "Шаблонний метод" (Template Method)
У суб'єкті є товстий натяк: "Перевіряти вимоги у кожному конкретному класі чи у базовому класі... один із цих способів є значно елегантнішим за інший".

Елегантний спосіб — це використання патерну Шаблонний метод:

У базовому класі AAForm метод execute(Bureaucrat const & executor) const робиться не віртуальним (або просто загальним). У ньому прописується загальна для всіх форм логіка перевірки:

Чи форма підписана? (Якщо ні -> throw AFormNotSignedException).

Чи грейд бюрократа дозволяє виконання? (Якщо ні -> throw GradeTooLowException).

Якщо перевірки пройшли успішно, метод execute всередині себе викликає чисто віртуальну (приватну або захищену) функцію, наприклад, executeAction() const:

C++
void AAForm::execute(Bureaucrat const & executor) const {
    if (!this->getIsSigned()) throw AAForm::AFormNotSignedException();
    if (executor.getGrade() > this->getGradeToExecute()) throw AAForm::GradeTooLowException();

    this->executeAction(); // Виклик поліморфної дії конкретного класу
}
Таким чином, вам не доведеться дублювати перевірку if (signed && grade) у трьох різних класах-нащадках. Нащадки описують тільки свою унікальну дію у функції executeAction.

4. Робота з потоками файлів (std::ofstream) та генерація випадкових чисел
Для реалізації логіки конкретних форм вам знадобляться стандартні бібліотеки C++:

Для ShrubberyCreationAForm: Використовуйте <fstream> (std::ofstream) для створення файлу та запису туди ASCII арту (текстових ялинок/дерев). Не забудьте закривати файли після роботи.

Для RobotomyRequestAForm: Для імітації 50% шансу успіху використовуйте генерацію випадкових чисел за допомогою функцій rand() та srand() з бібліотеки <cstdlib>. (Наприклад, rand() % 2 == 0). Для звуків дрилі достатньо просто вивести текст на кшталт * ВВВВВВВРРРРРР * через std::cout.
